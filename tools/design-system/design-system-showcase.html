Accordion Components

Collapsible content sections with ARIA-compliant interactive behavior and accessibility features.

Basic Accordion

Click headers to expand/collapse content sections.

Accessibility testing evaluates how well a website or application can be used by people with disabilities. This includes testing for screen reader compatibility, keyboard navigation, color contrast, and other WCAG guidelines.

Our automated testing uses axe-core to scan web pages for accessibility violations. It can detect issues like missing alt text, insufficient color contrast, improper heading structure, and missing form labels.

The system generates comprehensive HTML and JSON reports showing all accessibility violations found, organized by impact level (critical, serious, moderate, minor). Each violation includes detailed information and suggested fixes.

<div class="accordion"> <div class="accordion-item"> <h3 class="accordion-header"> <button class="accordion-trigger" aria-expanded="false"> <span class="accordion-title">Section Title</span> <span class="accordion-icon">▼</span> </button> </h3> <div class="accordion-content"> <p>Content goes here...</p> </div> </div> </div>

Report Accordion (Accessibility Violations)

Specialized accordion for displaying accessibility test results with impact levels.

Rule: image-alt

Description: Images must have alternative text to be accessible to screen reader users.

How to fix: Add descriptive alt text to all images using the alt attribute.

Rule: color-contrast

Description: Text must have sufficient contrast against its background.

How to fix: Use darker text colors or lighter backgrounds to achieve at least 4.5:1 contrast ratio.

Semantic Landmarks Guide

Proper use of HTML5 semantic elements for accessibility and SEO.

Semantic HTML5 Elements

<header>

Contains site branding, main navigation, or section headings. Use role="banner" for page header.

<nav>

Navigation links. Always include aria-label to distinguish multiple navs. Use role="navigation".

<main>

Primary content of the page. Only one per page. Must have unique content. Skip link target.

<section>

Thematic groupings of content. Should have a heading. Use aria-labelledby to reference heading.

<aside>

Sidebar content related to main content. Can contain navigation, ads, or supplementary info.

<footer>

Page or section footer. Contains copyright, links, contact info. Use role="contentinfo" for page footer.

💡 Accessibility Tip

Screen readers use these landmarks to navigate quickly through page content. Users can jump between landmarks using keyboard shortcuts, making proper semantic structure essential for accessibility.

Usage Guidelines

Best practices for implementing and customizing the design system in your projects.

Getting Started

1. Include the Design System

Add the design system CSS to your project:

<link rel="stylesheet" href="path/to/design-system.css">

2. Use Semantic HTML

Start with proper HTML structure and add design system classes:

<button class="btn btn-primary">Action</button> <div class="card">...</div>

3. Follow Accessibility Guidelines

Always include proper labels, alt text, and ARIA attributes for accessibility.

Customization

CSS Custom Properties

Override design tokens using CSS custom properties:

:root { --color-primary: #your-brand-color; --font-family-primary: 'Your Font', sans-serif; --space-4: 1.5rem; /* Adjust spacing */ }

Component Extensions

Extend existing components with additional styles:

.btn-custom { @extend .btn; @extend .btn-primary; border-radius: 2rem; /* Custom styling */ }

Do's and Don'ts

✅ Do

  • Use semantic HTML elements
  • Include proper ARIA labels
  • Test with keyboard navigation
  • Check color contrast ratios
  • Use consistent spacing tokens
  • Follow the established color hierarchy

❌ Don't

  • Override component styles arbitrarily
  • Use non-semantic div/span for buttons
  • Ignore focus indicators
  • Use color alone to convey meaning
  • Create inconsistent spacing
  • Break the established visual hierarchy

Accessibility Guidelines

Our design system is built with accessibility as a core principle, meeting WCAG 2.1 AA standards.

Key Features

  • Color Contrast: All text combinations meet WCAG 2.1 AA contrast requirements (4.5:1 minimum)
  • Keyboard Navigation: All interactive elements are keyboard accessible with visible focus indicators
  • Screen Reader Support: Semantic HTML and proper ARIA attributes for assistive technologies
  • Responsive Design: Mobile-friendly layouts that work across all device sizes
  • Clear Visual Hierarchy: Consistent heading structure and logical reading order
